# EA 全景地图工作笔记

> 日期：2026-05-26  
> 状态：工作草稿，不是最终规范  
> 目的：沉淀用户已确认的 EA 设计意图，为后续全景地图和 agent team 主线程 brief 提供依据。

---

## 1. 已确认的目标用户与痛点

EA v0.1 的第一批用户是项目发起人本人及团队成员，用户画像是材料学研究生和小型科研团队成员。

核心痛点：

1. 长期大型科研项目中，实验细节、样品编号、条件变化和阶段性判断容易丢失。
2. 用户不希望重复手动完成数据处理、制图、格式整理等低价值劳动。
3. 用户希望 EA 能结合科学原理、实验日志、实验内容、表征数据和相关文献，生成可参考的数据分析报告。
4. 用户希望 EA 能辅助判断项目下一阶段怎么做，但不能把推测包装成确定结论。

设计含义：

- EA 的长期价值应放在“研究主线记忆”和“可追溯科研整理”上，而不是单次数据处理。
- EA 必须支持跨天、多轮输入，而不是假设用户一次性提交完整实验包。
- EA 对用户的回答必须能引用已有实验记录、样品、报告和 provenance 来源。

---

## 2. 真实使用场景理解

真实使用方式是连续对话式输入：

```text
day1:
用户多次提交自然语言实验记录
→ EA 识别、结构化、请求审核
→ 保存为本地实验日志

day2:
用户询问“目前哪些样品比较好，适合做 Raman”
→ EA 从历史实验日志中检索候选样品
→ 给出依据和不确定性
→ 用户完成 Raman 表征后提交数据和样品编号
→ EA 关联样品、保存原始数据路径、分析数据、生成报告
```

设计含义：

- EA 不应只支持“一次性输入实验日志 + 数据文件”的线性流程。
- EA 必须支持增量式项目记忆。
- 样品推荐类回答必须说明判断依据，例如形貌、成功率、用户标注、已有表征结果，而不是只给结论。
- 数据文件可能晚于实验日志出现，因此 Experiment、Sample、Characterization File 之间需要允许后续关联。

## 3. 项目初始化原则

EA 项目初始化时，用户最低需要提供：

- 研究方向。
- 材料体系。
- 实验类型。

用户可以通过自然语言描述项目，不需要一次性填写完美表格。EA 应负责：

- 从自然语言中提取项目初始信息。
- 标记不确定或缺失字段。
- 及时向用户追问必要信息。
- 与用户交流确认最终项目初始信息。
- 将确认后的信息写入项目配置、项目规则卡和项目记忆。

常见表征手段不要求用户在初始化时一次性提供。EA 应在确认项目初始信息后，根据研究方向、材料体系和实验类型搜集相关资料和文献，向用户提供可能使用的常见表征列表。该列表应包含：

- 表征手段名称。
- 该表征对当前项目研究的作用。
- 适用阶段或适用问题。
- 推荐优先级。
- 是否为 EA 当前版本已支持的分析对象。

表征手段应允许后续修改：

- 用户可以随着项目进展增加、移除或改变表征手段。
- EA 不应把初始化时的表征列表视为固定项目边界。
- 新增表征手段时，应更新项目规则卡和项目记忆。
- 常见表征列表只用于帮助用户了解可能选择，不记录“用户已选择/暂不考虑/后续可能使用”等状态。

文件存储位置与本地目录结构应由 EA 合理设定和记忆：

- EA 应创建或维护清晰的本地项目目录结构。
- 用户不需要手动管理内部路径。
- 用户可以随时询问文件的具体存储位置。
- 用户可以要求 EA 调取相关文件。
- 项目目录结构创建采用“EA 设计 + 用户知悉”模式：EA 可以直接创建合理目录结构，创建后告知用户。
- 用户仍应保留修改目录或存储偏好的权利，但不要求用户具备计算机专业能力来设计目录结构。

项目初始化时，EA 应立即创建全局知识库目录和项目知识库目录。即使当前还没有文献条目，也应先建立目录结构，方便后续资料沉淀和路径稳定。

v0.1 只支持一个 EA 工作目录对应一个科研项目。多项目管理是后续版本方向，用于支持一个 agent 工作环境下同时管理多个研究项目。

## 4. 项目规则卡

EA 应为每个项目创建一份“项目规则卡”，作为长期项目处理和记忆的核心参考。

项目规则卡的作用：

- 帮助 EA 在长期项目中保持一致的处理规则。
- 让用户了解 EA 当前如何理解和管理项目。
- 为后续 agent team、子 skill 和报告生成提供稳定上下文。

项目规则卡至少应包含：

- 项目初始信息：研究方向、材料体系、实验类型。
- 样品编号规则。
- 样品评价标准。
- 默认报告语言。
- 默认数据处理偏好。
- 常见表征手段列表。
- 知识库策略。
- 原始文件管理策略。
- 用户已确认的关键规则。
- 规则版本与更新时间。

设计规则：

- 项目规则卡必须由 EA 和用户沟通后形成，不能由 EA 单方面最终确定。
- 规则卡可以后续更新，但每次重要变更都应记录变更原因、时间和用户确认状态。
- 项目规则卡中的关键规则必须逐项确认，不能只用一句整体确认替代。
- 关键规则包括样品编号规则、样品评价标准、知识库同步策略、原始文件管理策略、默认数据处理偏好、项目记忆写入规则和阶段冻结规则。
- 当用户的自然语言输入与项目规则卡冲突时，EA 应指出冲突并请求确认。
- 项目规则卡是规则参考，不是科学结论存放处。

## 5. 样品编号设计原则

v0.1 的 MoS2 test 样本可采用“时间 + 样品信息”的编号格式。用户提出的示例：

```text
260527-T01M2-E01
```

含义：

- `260527`：2026 年 5 月 27 日生长。
- `T01`：tube furnace 1，当天第一炉。
- `M2`：mica 2，第二片云母衬底。
- `E01`：experiment 1，实验组 1。

该格式可以继续优化，但必须满足：

- 用户能读懂。
- EA 和后续 agent 能稳定解析。
- 编号能在长期项目中保持唯一性和可追溯性。
- 编号能表达实验、炉次、衬底/样品之间的层级关系。

EA skill 的通用设计原则：

- 样品编号规则不应硬编码为 MoS2 专用格式。
- 在正式开始一系列实验前，EA 应根据用户对实验内容的初步描述，主动帮助用户设计样品编号格式。
- 编号格式必须经过用户讨论、修改和确认后才能投入使用。
- 确认后的编号规则应写入项目配置或项目记忆，并作为后续实验日志结构化的依据。
- 后续用户用自然语言提交实验记录时，EA 应根据已确认编号规则自动生成或配置样品编号。
- 如果自然语言信息不足以生成可靠编号，EA 必须标记不确定字段并请求用户补充，不能凭空补全。

## 6. 材料体系范围

MoS2 是 v0.1 的唯一材料体系，也是首个实景测试体系。v0.1 的 test 实验主要模拟 MoS2 CVD 生长场景，暂不包括转移、退火、掺杂、等离子处理等后处理。

长期目标：

- EA 最终版本应兼容多种材料体系。
- 与材料体系相关的规则应通过 material-system notes、知识文件、schema 扩展或子 skill 扩展实现。
- v0.1 不能把 MoS2 规则写死到 EA 核心架构里。
- test 实验只是用户模拟测试素材，不是 EA 的产品目标本身。

设计含义：

- MoS2 可以作为第一个垂直样板，把实验日志、Raman 分析、报告、记忆、provenance 闭环跑通。
- EA 核心对象仍应保持通用，例如 Project、Experiment、Sample、Material System、Process Condition、Characterization File。
- 构建 EA 架构的 agent 不应围绕 test 实验的具体内容设计系统。
- 参与用户模拟测试的 agent 可以知道 test 实验内容，用于模拟真实用户输入和评价 EA 输出质量。
- 全景地图需要区分“通用产品需求”和“测试样本内容”，避免 agent team 对测试集过拟合。

## 7. 建议的实景测试样本结构

v0.1 准备 `test-case-001` 作为正式实景测试样本，模拟真实用户输入，而不是整理成完美数据集。更多测试样本放到后续版本扩展。

测试样本的使用边界：

- test 样本用于模拟实际用户使用场景，判断 EA 是否满足用户需求。
- test 样本不应成为 EA 架构的硬编码依据。
- 构建 agent 只应看到通用需求、接口规范、验收标准和脱敏示例。
- 模拟用户 agent / evaluation agent 可以使用完整 test 样本和隐藏真值。
- 隐藏真值不应提供给被测 EA 或构建 agent。

每组测试样本建议包含：

1. `conversation.md`
   - 按时间顺序记录用户会怎么和 EA 对话。
   - 包括 3-6 段自然语言实验记录。
   - 包括至少 1 个后续查询，例如“哪些样品比较好”或“帮我找适合 Raman 的样品”。

2. `raw_data/`
   - 放 Raman 原始文件。
   - 文件格式尽量保留真实来源，如 `.txt`、`.csv`、`.xlsx`。
   - 文件名可以不完美，因为真实用户也可能用仪器默认命名或 U 盘拷贝命名。

3. `sample_file_mapping.md`
   - 记录 Raman 文件与样品编号的真实对应关系。
   - 这份文件可以作为测试真值，不一定模拟用户第一时间提供。

4. `expected_extraction.md`
   - 人工写出你期望 EA 至少识别出的字段。
   - 包括日期、实验组编号、样品 ID、材料体系、衬底、温度、保温时间、气氛/流速、观察结果、初步判断、不确定字段。

5. `expected_decisions.md`
   - 写出你认为哪些样品“比较好”以及理由。
   - 用于测试 EA 的样品推荐是否基于证据。

6. `notes_for_agent.md`
   - 记录特殊背景，例如某个缩写、某个炉次编号规则、某些描述只有组内成员懂。

测试样本内容建议覆盖：

- 成功样品、失败样品、半成功样品各至少一个。
- 至少一次用户表达模糊，例如“流速 60”但没说单位。
- 至少一次样品编号缺失或后补。
- 至少一次同一炉次内多个衬底结果不同。
- 至少一次 Raman 文件名和样品编号不完全一致。
- 至少一个数据文件需要用户确认 x/y 列。
- 至少一个实验记录中包含初步解释，但该解释不能直接进入 confirmed findings。

## 8. Raman 数据处理初步设计原则

Raman 分析 skill 不能靠 LLM 直接解释原始曲线，必须先由确定性脚本完成读取、清洗、预处理、检峰、绘图和元数据记录。

初步处理流程：

```text
读取文件
→ 识别表头、列名、数值范围、可能单位
→ 请求用户确认 Raman shift 列和 intensity 列
→ 展示默认参数
→ 请求用户确认参数
→ 生成原始谱图预览
→ 可选 cosmic ray/spike 检测
→ 可选 baseline correction
→ 可选 smoothing
→ 可选 normalization
→ peak detection / fitting
→ 输出图谱、峰表、处理后数据、metadata、warnings
→ LLM 只基于结构化结果生成审慎解释草稿
```

Raman 处理需要后续单独做文献调研，暂定参考来源包括：

- Nature Communications Materials 关于 Raman curve fitting 的综述，强调 baseline subtraction 和物理合理的峰模式初始假设。
- EPJ Techniques and Instrumentation 关于 Raman/IR 数据预处理的综述，涵盖谱轴对齐、cosmic ray/spike removal、baseline correction、smoothing、normalization。
- Nature Research Figure Guide，用于约束图表字体、颜色可访问性、导出格式和科研图表审美。

初步图表风格：

- 以 Nature 风格为参考，但 v0.1 输出以本地科研报告可读性优先，不强制投稿格式。
- 使用 Arial/Helvetica 类字体。
- 避免红绿组合和彩虹色图。
- 使用色盲友好配色，例如黑色、蓝色、橙色、蓝绿色。
- 坐标轴必须标注单位，Raman shift 使用 `cm^-1`。
- 原始数据、处理后数据、峰位标注必须视觉区分。
- 每张图必须能追溯到原始文件、处理参数和脚本版本。
- Raman 仪器导出的文件通常可能包含激光波长、功率、积分时间、物镜倍数等元数据；EA 应尽量自动提取并写入 metadata。
- 如果元数据缺失或解析不确定，EA 应在分析报告和 provenance 中显式标记。

## 9. 实验日志结构化审核

用户每次提交自然语言实验记录后，EA 必须先结构化并展示字段，待用户确认后再正式保存。

流程：

```text
用户提交自然语言实验记录
→ EA 解析并识别字段
→ 如存在不理解或缺失信息，EA 继续追问用户
→ EA 输出结构化字段供用户确认
→ 用户确认 / 修改 / 指出错误
→ EA 按用户意见修正
→ 用户确认后保存实验日志
```

设计规则：

- 不允许跳过用户确认直接把实验日志正式保存。
- 不允许以“低风险”为由先静默保存为正式记录。
- 如果用户指出结构化字段有误，EA 必须按用户要求修改后再次确认。
- 未确认内容可以临时处于工作草稿状态，但不能进入正式实验记录、项目进度或 confirmed memory。
- 该审核节点是 EA v0.1 的强制节点。
- 用户可以用自然语言整体确认结构化字段，例如“可以，保存”“没问题”“可以的”等明确审核通过表达。
- 用户不需要逐字段确认，但 EA 必须先展示结构化字段供用户检查。

## 10. 科学解释边界初稿

EA 可以做：

- 描述观察到的谱峰位置、峰强、峰宽和峰间距。
- 将结果与材料体系的常见 Raman 特征进行谨慎对照。
- 提出可能解释、待验证假设和下一步表征建议。
- 明确指出数据质量、处理参数、样品信息不足带来的限制。

EA 不可以做：

- 仅凭 Raman 单一数据断言材料相、缺陷浓度、载流子浓度或机制已经被证明。
- 把用户未确认的解释写入 confirmed findings。
- 忽略基底、激光波长、仪器参数、样品形貌和处理参数对 Raman 的影响。
- 为了给出完整报告而编造缺失实验条件、文献依据或结论。

必须要求用户确认或补充证据的内容：

- 材料相变判断。
- 层数、缺陷、应变、掺杂、电荷转移等强解释。
- 与工艺参数之间的因果关系。
- 可进入长期项目记忆或论文素材池的结论。

科学解释 reviewer skill 的证据等级列入后续版本设计考虑范围，暂不进入 v0.1。

一个不复杂的候选设计：

| 证据等级 | 含义 |
|---|---|
| `observation` | 用户观察或仪器/图像中直接看到的现象 |
| `processed_result` | 经过脚本处理得到的峰位、图表、峰表、统计结果等 |
| `literature_supported` | 有本地或公开文献支持的解释 |
| `hypothesis` | 基于当前证据提出但尚未验证的假设 |
| `user_confirmed` | 用户明确确认可写入项目记忆的结论 |

候选使用规则：

- reviewer skill 不负责生成新结论，只负责检查解释是否越界。
- 每条关键解释最多标一个主要证据等级，避免过度复杂。
- `hypothesis` 不能写入 confirmed findings。
- `user_confirmed` 必须来自用户审核，不得由 EA 自行赋予。

## 11. 样品质量判断标准

“比较好的样品”的判断标准应来自两个来源：

1. 科学资料和文献来源
   - 优先参考用户上传的本地文献。
   - 如果本地资料不足，EA 可以主动提出联网检索公开文献。
   - 检索到的参考资料应整理并存入本地知识库。

2. 用户设定和用户标注
   - 如果 EA 无法根据现有记录判断样品好坏，应主动向用户提问。
   - 用户对样品的正向/负向标注应被记录，并优先影响后续判断。

使用规则：

- EA 建立样品判断标准后，必须整理给用户确认。
- 未经用户确认的标准只能作为草稿标准，不能自动用于长期决策。
- 样品推荐必须给出证据来源，例如形貌、尺寸、覆盖率、污染、用户标注、Raman 结果或文献依据。
- 样品推荐应包含不确定性说明，不能只给出排名。
- v0.1 不应把 MoS2 test 实验中的样品好坏标准直接写入 EA 架构。
- 对任何新项目，EA 应通过用户沟通、用户标注、项目文献和材料体系知识建立该项目自己的样品评价标准。
- 如果标准不足，EA 应主动向用户提问，而不是套用 test 样本答案。

## 12. 项目记忆定位

EA 的长期记忆定位是：

```text
研究主线 + 论文素材池
```

设计含义：

- 记忆不是普通聊天历史，而是有证据等级和来源链条的研究资产。
- confirmed findings 只保存用户确认后的发现。
- open questions 保存待验证假设。
- failed attempts 保存失败实验、异常样品和无效工艺窗口。
- decision log 保存用户做出的方向性判断。
- 后续论文写作需要能从记忆中抽取：实验路线、关键样品、有效条件、失败路径、表征证据和已确认结论。

## 13. 项目主体记忆写入边界

EA 必须严格区分：

- EA 给出的建议。
- 用户明确采纳的项目决策。
- 用户明确描述已经完成的项目进度。

核心规则：

- EA 的“下一步实验建议”不能直接写入项目决策。
- EA 的建议应有单独审核状态，不能默认进入项目主体记忆。
- 只有用户明确表示“我采用你的建议”“我计划下一步做……”等项目决策类表达时，相关内容才能写入 decision log。
- 只有用户明确表示已经完成某项实验、表征或数据提交时，EA 才能把该事项作为项目进度写入项目记忆。
- 例如，EA 不能因为自己建议做 Raman，就把“已进行 Raman 表征”写入项目进度；必须等用户确认完成 Raman 或提交 Raman 数据。
- 项目主体记忆中的内容必须经过用户审核或由用户明确表达确认，不能由 EA 自行替用户决策。

建议记录的可存放位置：

- 报告后的建议区。
- next-step suggestion 草稿记录。
- open questions 或待验证假设。
- 待用户确认的 decision candidate。

下一步实验建议的审核状态采用简单枚举：

| 状态 | 含义 |
|---|---|
| `draft` | EA 生成的建议，用户尚未确认 |
| `accepted` | 用户明确表示采纳该建议或计划按该方向执行 |
| `modified` | 用户基于 EA 建议做了修改后形成自己的计划 |
| `rejected` | 用户明确拒绝该建议 |

转换规则：

- `accepted` 或 `modified` 的建议仍不能自动写入项目决策。
- 只有当用户自然语言明确表达“我决定/计划/采用/下一步做……”时，EA 才能把对应内容转换为正式 decision log 条目。
- 不需要为项目进度设置复杂状态机。EA 保存每一个用户确认的项目进度事件，并在用户提问时基于事件记录准确描述项目进度。

但这些内容在用户确认前不能成为：

- confirmed findings。
- decision log 中的正式项目决策。
- project progress 中的已完成进度。
- 论文素材池中的用户确认结论。

## 14. 阶段性冻结

EA 需要支持“阶段性冻结”功能，用于在某个项目阶段锁定一版项目状态。

触发方式：

- EA 在判断项目达到阶段节点时，可以提出冻结建议，并询问用户意见。
- 用户也可以主动要求冻结当前项目状态。
- EA 不应频繁提出冻结建议，避免打扰用户。

EA 可主动建议冻结的典型节点：

- 阶段性成果被用户确认，例如经过一段时间实验和表征后，用户确认某一材料、样品、工艺或表征结论已经成立。
- 项目路线需要明显转向或重启，例如一段研究被证明失败、存在明显错误、实验体系需要调整，或前一阶段结果需要归档后另起阶段。
- 完成阶段报告、阶段复盘或论文素材整理后，用户需要锁定当前项目状态。

冻结内容可以包括：

- 项目规则卡版本。
- 已确认实验记录索引。
- 已确认样品列表。
- 已确认报告和处理结果。
- confirmed findings。
- open questions。
- failed attempts。
- decision log。
- 论文素材池当前版本。
- provenance 索引。

设计规则：

- 阶段性冻结必须经过用户确认。
- 冻结不是阻止后续项目继续发展，而是保存一个可追溯的阶段快照。
- 冻结后的新实验、新数据和新解释应进入后续版本，不应静默改写已冻结状态。
- 如需修正冻结版本中的错误，应记录修正原因、修正时间和用户确认。
- 阶段性冻结采用混合快照策略：生成冻结 manifest 和 hash 指纹，并复制关键文本状态文件；大型原始数据和处理结果通过不可变路径和 hash 指向，不默认重复完整复制。
- 关键文本状态文件包括项目规则卡、项目记忆、decision log、confirmed findings、open questions、failed attempts、论文素材池索引和 provenance 索引。
- 该策略优先保证长期准确性、可验证性和存储效率。

冻结 manifest 至少记录：

- freeze_id。
- created_at。
- freeze_reason。
- user_confirmation。
- included_records。
- file_paths。
- file_hashes。
- source_labels。
- snapshot_notes。

## 15. 项目进度事件

项目进度不使用复杂状态机，而使用用户确认的进度事件记录。

项目进度事件最小字段：

| 字段 | 含义 |
|---|---|
| `progress_id` | 进度事件唯一 ID |
| `recorded_at` | EA 记录该事件的时间 |
| `occurred_at` | 用户描述的实际发生时间，未知则留空 |
| `user_original_text` | 用户原始表述 |
| `ea_summary` | EA 整理后的简洁进度描述 |
| `event_type` | 例如 experiment、characterization、data_upload、analysis、report、decision |
| `related_experiments` | 关联实验编号 |
| `related_samples` | 关联样品编号 |
| `related_files` | 关联文件或数据 |
| `source_refs` | 可追溯来源 |
| `uncertainties` | 不确定或待补充信息 |

设计规则：

- 进度事件必须来自用户明确表述或用户提交的文件/数据。
- EA 不能把自己的建议写成项目进度。
- 用户提问项目进度时，EA 应基于进度事件按时间和主题组织回答。

## 16. 论文素材池组织

后续论文素材池应按以下维度组织：

- 结果图。
- 实验条件。
- 关键结论。
- 失败路径。
- 可引用表述。

设计规则：

- 论文素材池不是 confirmed findings 的替代品。
- 进入论文素材池的内容必须保留来源链条，例如实验记录、数据文件、处理结果、报告和用户审核记录。
- “可引用表述”必须区分草稿表达、用户确认表达和待验证表达。
- 失败路径也应被保留，因为它能帮助后续论文写作解释实验优化过程和排除路线。

## 17. 报告定位

v0.1 的原始数据分析报告主要给用户自己看。

设计含义：

- 报告应重视清晰、完整、可追溯，而不是直接模仿论文正文。
- 默认报告可以更像“分析记录 + 解释草稿”。
- 默认 Raman/原始数据分析报告不包含下一步实验建议。
- 报告返回后，如果用户追问下一步怎么做，EA 应能基于报告、实验数据、项目记忆和文献依据给出可靠建议。
- 如果用户后续明确要求组会、论文、补充材料格式，再由 report_generation skill 转换表达风格和格式。
- v0.1 默认只支持中文报告。
- 后续最终版本需要支持中文和英文，但这不作为 v0.1 必做项。

## 18. 原始文件管理建议

建议 EA 采用：

```text
复制进入项目 raw 目录作为受控只读副本 + 保留原始来源路径和文件指纹
```

理由：

- 只记录 U 盘或桌面路径容易在长期项目中失效。
- 复制到项目目录更利于备份、迁移、复现实验分析和 agent team 协作。
- 原始数据仍然必须保持只读，不允许覆盖、重写或原地清洗。
- provenance 中应记录导入时间、原始路径、项目内 raw 路径、文件 hash、文件大小和导入方式。

设计规则：

- `raw/` 中的文件视为不可变原始副本。
- 数据处理输出必须写入 `processed/`，不能写回 `raw/`。
- 如用户担心空间占用，后续可支持“仅记录路径”模式，但 v0.1 建议默认使用受控副本模式。
- 如果同一文件重复导入，EA 应通过 hash 检测重复，避免制造多个不必要副本。

## 19. 文献与本地知识库策略

EA 允许主动联网检索公开文献。

项目初始化时应立即创建知识库目录，包括全局知识库和项目知识库。

知识库应分为：

1. 全局知识库
   - 存放可跨项目复用的通用科学知识、材料体系基础资料、表征方法说明、数据处理方法和通用文献笔记。

2. 项目知识库
   - 存放与当前研究项目直接相关的文献、资料、用户笔记、项目特定解释依据和项目特定样品评价标准。

设计含义：

- 全局知识库服务长期复用。
- 项目知识库服务当前项目的溯源、解释和决策。
- 项目报告和项目记忆引用文献时，应优先说明引用来自项目知识库还是全局知识库。
- 全局知识库和项目知识库之间允许自动同步。
- 如果用户明确要求某些资料“不需要同步”，EA 必须尊重并标记为不同步。
- 被用户标记为“不需要同步”的内容，后续只有在用户明确要求恢复同步时才能恢复同步。
- 自动同步不应覆盖项目特定笔记；跨库复用时应保留原始来源和同步记录。

触发条件：

- 用户明确要求参考文献。
- EA 判断现有知识不足以支撑解释、处理参数或样品评价标准。
- EA 需要为材料体系、Raman 特征、样品质量标准或下一步实验建议建立依据。

存储规则：

- 检索到的参考资料应存储到本地知识库。
- 本地知识库应同时保存可获得的 PDF/HTML 全文、文献条目、摘要、链接和用户/EA 笔记。
- 每条文献或资料需要记录来源、访问时间、标题、作者、年份、链接、摘要、与当前项目的相关性。
- 如果只能访问摘要或元数据，EA 必须标记全文不可用。
- 如果 EA 无法下载全文，应向用户报告并请求用户下载后上传。

安全与边界：

- EA 可以协助打开网页、定位资料和整理下载需求。
- 如果涉及机构账号、个人账号或付费资料，EA 不应保存账号密码，也不应绕过访问权限。
- 未来如整合自动下载 skill，应以用户授权、合法访问和本地凭据安全为前提。

## 20. 待补充信息表

EA 应维护一个“待补充信息表”，用于长期追踪项目中的缺失信息和未完成审核。

可追踪内容包括：

- 缺失或不确定的样品编号。
- 缺失的实验条件或单位。
- 待上传的 Raman 或其他表征文件。
- 待确认的数据列和处理参数。
- 缺失的仪器元数据。
- 待用户审核的实验记录、科学解释、报告和记忆写入。
- EA 建议补充的文献或背景资料。

设计规则：

- 每条待补充项应包含来源、关联实验/样品、优先级、状态、创建时间和解决时间。
- EA 在回答用户问题或开始相关工作流前，应检查是否存在会影响结论可靠性的未解决待补充项。
- 待补充信息不能静默影响结论，必须在报告或回答中显式说明。
- 默认情况下，EA 不应在每次对话中主动展示完整待补充信息表。
- 用户主动询问时，EA 应展示待补充信息表或相关条目。
- 当某个待补充项会直接影响当前回答或报告可靠性时，EA 应只提示与当前问题相关的缺失信息，避免冗余内容增加用户阅读成本。

## 21. 回答与来源策略

EA 回答历史查询时，默认应附带来源。

适用场景：

- 用户询问过去实验记录。
- 用户询问样品表现或样品推荐。
- 用户询问项目进度。
- 用户询问某个结论、报告或数据处理结果。
- 用户询问下一步建议的依据。

设计规则：

- 默认来源应使用简短括号格式，不增加过多阅读负担。
- 来源应指向实验记录、报告、原始数据、处理结果、项目记忆、知识库条目或 provenance 记录。
- 简短来源应优先显示用户友好的实验编号、样品编号、报告标题或数据名称，而不是内部文件名。
- EA 内部必须保留从用户友好来源标签到真实文件路径、记录 ID 或 provenance ID 的准确映射。
- 如果用户根据简短来源要求提供文件位置、打开文件或提取文件内容，EA 必须能准确定位对应文件。
- 如果用户继续追问“依据/来源/展开”，EA 再展开更完整的来源链条。
- 如果回答包含不确定性，EA 应说明不确定性来自缺失数据、未经审核内容、文献不足或数据质量问题。

来源格式示例：

```text
目前更适合优先表征的是 260527-E01-M02，因为该样品在光镜下形貌更规则、边界更清晰（来源：exp-20260527-001）。
```

## 22. Agent team 初步组织建议

建议采用“主线程 orchestrator + 专项子 agent/skill”的方式。

主线程负责：

- 持有 EA 全景地图。
- 判断任务是否偏离 EA 边界。
- 管理跨模块接口。
- 决定哪些节点必须请求用户审核。
- 整合子 agent 输出。

建议角色：

1. Architecture agent
   - 负责目录结构、schema、接口、工作流协议。

2. Raman/script agent
   - 负责确定性数据读取、处理、绘图、峰识别脚本。

3. Scientific language/reviewer agent
   - 负责科学解释边界、证据等级、过度推断检查。

4. Memory/provenance agent
   - 负责项目记忆结构、审核记录、溯源记录和检索规则。

5. Documentation/test agent
   - 负责实景测试样本、验收标准、用户工作流说明。

6. Simulated user / evaluation agent
   - 负责使用 test 样本模拟真实用户输入。
   - 持有隐藏真值，用于评价 EA 是否正确识别、追问、存储、关联和回答。
   - 不参与 EA 架构实现，避免测试答案泄漏给构建 agent。

执行原则：

- 架构和 schema 先行。
- Raman 脚本和测试样本可以并行。
- 科学解释必须在结构化数据和处理结果之后生成。
- memory 写入必须在用户审核之后发生。
- 测试样本内容与 EA 构建上下文隔离，防止过拟合。

最终交给 agent team 主线程的全景地图建议拆成“一个入口 brief + 四份专项文档”：

1. `EA_AGENT_TEAM_BRIEF.md`
   - 主入口，说明 EA 北极星、不可违反原则、工作分配方式和文档索引。

2. `EA_PRODUCT_CHARTER.md`
   - 定义目标用户、非目标、产品边界、科研审慎原则和用户体验风格。

3. `EA_ARCHITECTURE_MAP.md`
   - 定义核心实体、目录结构、schema、skill 边界、数据流和审核节点。

4. `EA_TEST_PROTOCOL.md`
   - 定义用户模拟测试、隐藏真值、评估方式和防过拟合规则。

5. `EA_AGENT_TEAM_PROTOCOL.md`
   - 定义主线程和子 agent 职责、上下文隔离、任务分配、review 和交付规则。

理由：

- 主线程可以先读 brief 建立全局方向。
- 子 agent 可以只读自己相关的专项文档，降低上下文负担。
- 测试协议独立出来，有利于隐藏 test 真值，避免构建 agent 过拟合。
- 架构地图独立出来，有利于后续直接驱动实现。
