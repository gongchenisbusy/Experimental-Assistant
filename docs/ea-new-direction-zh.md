# EA 新方向整理：Agent-Native 科研 Skill 系统

> 历史参考说明：本文记录 EA 从独立 Web 应用设想到 agent-native skill 系统的转向过程。当前项目级设计以 `EA_PROJECT_DESIGN.md` 为准，v0.2 工作计划以 `EA_V0_2_WORK_PLAN.md` 为准。本文中的“初版”“v0.1”“Raman”表述应按历史阶段理解，不再作为项目整体目标。

> 版本：v0.2-direction-draft  
> 日期：2026-05-11  
> 目的：记录 EA 从独立本地应用转向 Agent-native skill 系统后的项目方向、设计原则和初版实现边界。本文档面向人类与 coding agent，后续实现、讨论和拆分任务时应优先参考。

---

## 1. 核心转向

EA 原先的设想是一个独立的本地 Web 产品：

- React / Vite 前端
- FastAPI 后端
- 本地文件系统 + SQLite
- 模块化科学技能
- 独立 UI 承载实验日志、数据处理、报告和工作流审核

新的方向是：

**EA 不再优先开发独立 UI 和完整本地应用，而是封装成一个可以依赖 Claude Code、Codex 等现有 agent 运行的综合型科研 skill。**

也就是说，EA 的首版形态从“应用程序”变为“Agent-native skill 包”。用户通过现有 agent 与 EA 交互，EA 负责组织科研流程、调用子技能、读写本地项目数据、生成报告、维护项目记忆和记录溯源。

这个转向的目标不是降低 EA 的产品性，而是更快验证真正关键的问题：

> 实验科学研究者是否需要一个懂实验流程、懂数据处理边界、懂科研表达方式，并能长期维护本地研究记忆的 AI 实验助手？

---

## 2. 新架构定位

### 2.1 EA 是什么

在新方向下，EA 是一个：

- 本地优先的科研工作流 skill
- 面向实验科学研究者的 agent 扩展能力
- 由一个总控 skill 和多个子 skill 组成的科研助手系统
- 依赖现有 agent 执行交互、文件操作、脚本运行和报告生成
- 通过本地文件系统维护实验记录、处理结果、报告、记忆和溯源

一句话定义：

**EA 是一个 Agent-native 的实验科研助手 skill，帮助研究者把实验日志、原始数据、数据处理、科学解释、人工审核和项目记忆组织成可追溯的本地研究系统。**

### 2.2 EA 不是什么

EA 不是：

- 独立 Web 应用的首版实现
- 完全自主的科学家
- 简单的一段长 prompt
- 只会生成报告的脚本
- 实验室设备控制器
- 替代研究者进行科学判断的自动决策系统
- 会修改原始数据的自动处理工具

---

## 3. 架构原则

### 3.1 不做一个巨大提示词

EA 不应该退化成一个越来越长、越来越难维护的 prompt。它应该是一个结构化 skill 包，至少包含以下层次：

| 层级 | 职责 |
|---|---|
| EA Orchestrator | 总控流程，判断用户意图，组织任务计划，调用子 skill，发起审核请求 |
| Experiment Log Skill | 将自由文本实验日志结构化，提取样品、材料体系、工艺条件和关联文件 |
| Raman Analysis Skill | 读取 Raman 原始数据，绘图，检峰，输出峰表和审慎分析草稿 |
| Generic Data Plot Skill | 读取 CSV/TXT/XLSX，预览列，用户确认后绘图和保存元数据 |
| Report Skill | 基于已确认实验记录、处理结果和解释生成 Markdown/HTML 报告 |
| Memory Skill | 将用户确认后的结论、开放问题和实验线索写入项目主线记忆 |
| Provenance Skill | 记录输入文件、处理参数、技能版本、输出文件和用户审核决策 |

### 3.2 文件系统是首版的产品界面

由于首版不做独立 UI，本地文件结构本身就是 EA 的“隐形 UI”。它必须足够清晰、稳定、可读，让用户和 agent 都能长期理解。

建议最小项目结构：

```text
ea-project/
├── EA_PROJECT.md
├── experiments/
├── raw/
├── processed/
├── reports/
├── memory/
└── provenance/
```

各目录职责：

| 路径 | 职责 |
|---|---|
| `EA_PROJECT.md` | 项目总览、材料体系、研究目标、当前状态 |
| `experiments/` | 结构化实验记录，建议 Markdown + JSON/YAML front matter |
| `raw/` | 原始实验与表征数据，只读，不允许覆盖或修改 |
| `processed/` | 处理后的 CSV、图表、峰表、统计结果 |
| `reports/` | 已生成的实验报告、阶段报告、分析报告 |
| `memory/` | 项目主线记忆，包括已确认发现、开放问题、失败尝试等 |
| `provenance/` | 每次工作流运行的溯源记录和审核决策 |

### 3.3 计算尽量确定，解释必须审慎

Raman 等数据分析不应直接让 LLM 读取原始数据后自由解释。更可靠的流程是：

```text
原始 Raman 数据
→ Python 脚本读取与处理
→ 输出图谱、峰表、处理参数、警告和元数据
→ LLM 基于结构化结果生成审慎解释
→ 用户审核解释
→ 写入报告和项目记忆
```

确定性脚本负责：

- 文件读取
- 数据清洗
- 单位识别或提示
- 平滑、归一化、基线校正等可配置处理
- 峰识别
- 绘图
- 输出机器可读元数据

LLM 负责：

- 解释处理结果
- 提示可能的科学含义
- 标注不确定性
- 整理报告语言
- 给出下一步实验建议

但 LLM 不能把未经用户确认的假设写成已确认事实。

## 4. 核心竞争力

EA 的竞争力不在于通用 AI 能力，而在于科研场景理解。

它必须从实验科学研究者的真实工作出发设计，包括：

- 实验记录习惯
- 样品编号和材料体系管理
- 工艺参数记录方式
- 表征数据处理流程
- 科学解释的证据边界
- 报告阅读习惯
- 图表和数据文件的专业规范
- 失败实验、小观察和开放问题的长期沉淀

EA 的目标用户是实验科学研究者，因此判断标准不是“回答得像一个聪明聊天机器人”，而是：

- 是否可靠
- 是否准确
- 是否可复现
- 是否符合科研流程
- 是否节省整理时间
- 是否让研究者愿意长期使用
- 是否让用户感觉它真的理解实验科研，而不是泛 AI 套壳

## 5. 科研审美

这里的“科研审美”不是单纯的视觉风格，也不是把界面做得像论文或实验室海报。它更接近一种专业感：

> 研究者看到 EA 的输出、流程、命名、图表和语言后，会觉得它克制、准确、可信、可复用，并且尊重科研证据边界。

科研审美应被工程化为可执行规则。

### 5.1 图表审美

- 图表必须有清晰标题、坐标轴、单位、图例或必要标注
- Raman 位移必须标注为 `cm^-1`
- 峰位标注应克制，不遮挡数据
- 不使用花哨配色、夸张阴影或装饰性元素
- 原始数据、处理后数据、拟合结果应明确区分
- 如果进行了平滑、归一化或基线校正，应在图注或元数据中说明

### 5.2 报告审美

报告应符合科研人员阅读习惯，建议结构：

1. 实验背景
2. 样品与工艺条件
3. 原始数据来源
4. 数据处理方法
5. 主要观察结果
6. 计算或检测结果
7. 可能解释
8. 不确定性与限制
9. 下一步建议
10. 溯源记录

### 5.3 语言审美

推荐使用：

- “可能表明”
- “与……一致”
- “可能与……相关”
- “需要进一步确认”
- “在当前数据范围内”
- “该解释仍需结合其他表征验证”

避免使用：

- “这证明了”
- “这确认了”
- “这个机制是确定的”
- “毫无疑问”
- “可以完全说明”

### 5.4 流程审美

- 该自动的步骤自动完成，例如读取、绘图、整理元数据
- 该让用户判断的步骤必须停下来，例如字段确认、参数确认、科学解释确认、记忆写入确认
- 失败实验和异常数据不应被忽略，而应被结构化记录
- 用户修改过的内容必须被视为更高优先级事实

## 6. 初版设计

初版目标是跑通一个完整但极小的科研闭环：

```text
用户提交实验日志
→ EA 结构化实验日志并保存到本地
→ EA 请求用户审核结构化字段
→ 用户提交 Raman 原始数据
→ EA 读取数据、建议处理方案和参数
→ EA 请求用户确认数据列和处理参数
→ EA 运行 Raman 分析
→ EA 生成图谱、峰表、处理后数据和草稿解释
→ EA 请求用户审核科学解释
→ EA 生成分析报告
→ EA 请求用户确认是否写入项目主线记忆
→ EA 将已确认报告、结论和溯源保存到本地项目
```

### 6.1 初版必须保留的审核节点

即使没有 UI，审核也必须通过 agent 对话完成，并写入本地文件。

| 节点 | EA 必须做什么 |
|---|---|
| 实验字段审核 | 展示结构化实验记录，请用户确认、编辑或拒绝 |
| 数据列审核 | 展示检测到的数据列，请用户确认 x/y 列含义 |
| 参数审核 | 展示平滑、归一化、基线校正、检峰参数 |
| 科学解释审核 | 要求用户确认、修改或拒绝分析解释 |
| 记忆写入审核 | 写入长期项目记忆前必须请求用户同意 |

### 6.2 审核结果必须文件化

示例：

```yaml
review_status: user_confirmed
reviewed_at: 2026-05-11
reviewed_by: user
decision: accepted_with_edits
notes: 用户确认峰位解释，但删除了关于应力变化的推断。
```

审核结果不应只留在对话上下文里。

## 7. 项目主线记忆设计

EA 的记忆不应该是不断追加的流水账。它必须分层管理。

建议结构：

```text
memory/
├── project-summary.md
├── confirmed-findings.md
├── open-questions.md
├── failed-attempts.md
├── material-system-notes.md
└── decision-log.md
```

各文件职责：

| 文件 | 职责 |
|---|---|
| `project-summary.md` | 项目目标、当前阶段、材料体系、关键进展 |
| `confirmed-findings.md` | 用户确认后的发现或结论 |
| `open-questions.md` | 尚未解决的问题、需要后续实验验证的假设 |
| `failed-attempts.md` | 失败实验、无效参数、异常结果和可能原因 |
| `material-system-notes.md` | 针对材料体系的长期观察和经验性备注 |
| `decision-log.md` | 用户做出的关键判断、方向调整和取舍 |

关键规则：

- 已确认发现、待验证假设、失败尝试和用户决策必须分开
- 未经审核的解释不能写入 `confirmed-findings.md`
- 用户修改后的解释优先于 agent 原始输出
- 每条记忆应能追溯到实验记录、报告或 provenance 文件

## 8. 后续迭代：领域知识增强与置信度校准

这部分属于初版闭环完成后的后续方向，不应阻塞 v0.1。

### 8.1 问题

初版中，LLM 可能基于结构化数据生成专业解释，但在特定材料体系或表征方法上仍可能：

- 过度推断
- 混淆机制
- 忽略实验条件限制
- 把相关性写成因果关系
- 把单一表征结果解释得过于确定

### 8.2 不建议一开始做模型微调

除非后续有明确数据集、评估标准和训练资源，否则不应过早做模型微调。更现实的路径是：

1. 领域知识文件增强
2. 评审型 agent 或 reviewer skill
3. 基于评审反馈迭代 EA skill

### 8.3 路径一：领域知识文件增强

将可靠教材、综述、实验规范、仪器说明、领域论文中的稳定知识蒸馏为 Markdown 指南，供 EA 分析时引用。

示例：

```text
knowledge/
├── raman/
│   ├── raman-analysis-principles.md
│   ├── raman-common-artifacts.md
│   └── mos2-raman-notes.md
└── reporting/
    └── cautious-scientific-language.md
```

这些知识文件应强调：

- 适用条件
- 禁止过度解释的边界
- 常见假阳性
- 需要联合验证的情形
- 推荐报告语言

### 8.4 路径二：评审型 agent

构建一个 reviewer skill，不直接生成最终分析，而是检查分析结论并标注：

- 证据等级
- 置信度
- 是否过度断言
- 是否需要更多表征支持
- 是否有专业风险
- 是否符合报告语言规范

评审结果返回给其他协作 agent，用于优化提示词、处理参数、报告模板和知识文件。

### 8.5 推荐迭代顺序

1. 先跑通 v0.1 闭环
2. 收集真实或真实风格样例
3. 建立人工评审标准
4. 写领域知识 Markdown
5. 加 reviewer skill
6. 建立少量回归样例
7. 再考虑更复杂的蒸馏、检索或模型训练

---

## 9. 初版最应该验证的问题

EA v0.1 最重要的验证问题是：

1. 实验日志能否稳定变成结构化记录？
2. Raman 原始数据能否稳定生成图谱、峰表、处理后数据和报告？
3. 用户是否愿意参与审核？
4. 用户是否愿意把确认后的结果写入项目记忆？
5. 一周后，用户能否通过 EA 找回之前的实验线索？
6. EA 的报告是否真的比手动整理节省时间？
7. 科研人员是否觉得 EA 可靠、克制、准确，而不是泛 AI 聊天工具？

---

## 10. 对后续 agent 的执行要求

后续任何 agent 在实现 EA 时必须遵守：

1. 不要把 EA 做成一个巨大 prompt
2. 不要优先开发独立 UI
3. 不要修改原始数据
4. 不要跳过用户审核节点
5. 不要把假设写成已确认事实
6. 计算处理优先使用确定性脚本
7. LLM 解释必须基于结构化输出和明确证据边界
8. 所有输出必须有元数据和溯源
9. 项目记忆必须分层，不允许变成流水账
10. 科研审美必须体现在图表、报告、语言、命名和流程中
