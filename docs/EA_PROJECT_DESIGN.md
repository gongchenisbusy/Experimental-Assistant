# Experimental Assistant (EA) 项目设计

> 状态：项目级当前设计稿  
> 日期：2026-06-30  
> 用途：定义 EA 的长期项目目标、核心设计原则和稳定架构语言。版本目标请写入版本计划，不写进本文件作为项目整体目标。

## 1. 项目定位

Experimental Assistant（EA，实验助手）是面向实验科学研究者的本地优先科研工作系统。它帮助研究者把实验日志、样品、原始数据、处理结果、科学解释、报告、项目记忆和证据链组织成一个可追踪、可复用、可长期演化的研究空间。

EA 的核心不是某一种表征方法，也不是某一个前端形态，而是一个能持续沉淀研究过程的工作流内核：

```text
研究输入
→ 结构化记录
→ 数据导入与处理
→ 证据化解释
→ 用户确认
→ 报告与项目记忆
→ 后续检索、复盘和迭代
```

首批场景可以从材料科学、MoS2、Raman 数据进入，但这些只是验证入口，不是 EA 的项目边界。

## 2. 长期目标

EA 的长期目标是成为实验研究者的本地科研副驾驶：

- 降低实验记录整理、数据处理、报告撰写和证据追踪的负担。
- 让实验过程中的小观察、失败尝试、开放问题和阶段性判断不再散落丢失。
- 让数据分析过程可复查，参数、脚本、输入、输出和用户确认都有记录。
- 让科研解释保持证据分层，避免把猜测、建议或模型生成内容伪装成已确认结论。
- 支持不同材料体系、不同表征方法、不同研究阶段逐步接入同一套项目记忆和溯源框架。
- 支持模块化子 skill 接入，让文献库、绘图、Raman、PL、XRD、IR、显微图像、电学和其他专业能力都能遵守同一套项目结构。

## 3. 目标用户

EA 面向需要长期积累实验线索的实验科学研究者，尤其是日常处理以下材料的人：

- 实验日志、样品编号、工艺条件和观察记录。
- Raman、PL、AFM、XRD、SEM/TEM、电化学等表征数据。
- 数据处理脚本、图表、报告和组会材料。
- 文献笔记、研究假设、阶段决策和待验证问题。

EA 的体验标准不是“回答像通用聊天机器人”，而是研究者是否觉得它可靠、克制、记得住上下文，并且能帮助自己更快回到真实研究问题。

## 4. 核心设计原则

### 4.1 研究者保留判断权

EA 可以整理、计算、比较、提示和生成草稿，但关键判断应由研究者确认。用户确认不是形式主义，而是区分“系统建议”和“项目事实”的机制。

### 4.2 数据资产受保护

原始数据是研究资产。EA 应以受控副本、哈希、元数据和只读约束保护原始文件，并把处理结果写入独立输出区。

### 4.3 计算可复现，解释可追溯

数据处理尽量由确定性脚本完成，并记录参数、版本、输入和输出。语言解释应基于结构化结果和明确证据，不直接替代数据处理。

### 4.4 项目记忆分层

EA 的记忆不是一个混合聊天记录。它至少区分：

- 已确认发现
- 用户决策
- 项目进展
- 开放问题
- 失败尝试
- 系统建议
- 待用户复核内容

不同类型的记忆有不同写入条件、来源要求和后续用途。

### 4.5 工作流可扩展

EA 应允许新表征、新材料体系、新报告模板和新分析技能逐步加入。扩展能力应接入统一的项目、样品、raw、processed、report、memory 和 provenance 结构，而不是形成孤立脚本集合。

### 4.6 用户体验要轻

EA 的流程应减少低价值整理劳动，而不是制造过多仪式化确认。重要节点需要确认；显而易见的文件类型校验、格式错误和基本输入检查应作为普通工程正确性处理，不应写成项目原则。

### 4.7 来源支持的主动建议

谨慎不等于沉默。EA 可以基于项目背景、本地文献库、公开数据库、已登记 reference 或用户提供的资料，主动建议可能适用的参数、谱峰归属、处理窗口、模型假设和下一步实验。此类建议必须记录来源、适用条件、置信度和替代解释。

当建议参数会改变数据处理结果时，EA 应区分 `reported_by_user`、`source_suggested` 和 `user_confirmed_source_suggested` 等来源状态。来源支持的候选参数可以进入报告讨论；进入确定性处理、拟合约束、项目记忆或正式结论前，必须有可追溯 reference、适用性说明和必要的用户确认。

因此，EA 的边界应按证据质量和应用层级定义：不使用无来源常数，不静默套用会改变结果的参数，不把单一谱图候选解释写成证明性结论，不在缺少适用性说明和审核记录时把候选值升级为处理参数或长期记忆。背景明确、来源可靠时，EA 应该主动把可查证的科学参数、化学态候选和替代解释带入讨论。

若某个命令或 workflow 写明“不自动查找/不自动应用”，其含义是该确定性步骤不应未经用户确认地联网、下载、解析全文或改变处理参数；这不应被理解为 EA 不能在用户确认的检索、本地文献库、项目 reference 或用户提供资料基础上主动整理候选参数和解释。

### 4.8 面向公开用户初始化

EA 最终需要公开给陌生用户使用，因此项目设计不能默认采用开发者本机已有配置。Zotero 地址、学校认证路径、浏览器 profile、本地文献库文件夹、测试数据路径和用户私有账号信息，都必须通过初始化向导、项目配置或用户确认获得。

开发者本机配置只能用于 local integration test，不能写成产品默认值。需要登录或机构认证时，EA 应提示用户在自己的浏览器环境中手动完成，不保存凭据、不假设某个学校或某台机器的认证路径。

详细约束见 `EA_PUBLIC_RELEASE_INITIALIZATION.md`。

## 5. 系统形态

EA 当前优先采用 agent-native 形态：运行在 Codex、Claude Code 等 agent 环境中，通过本地文件系统、Python 工具和可复用 skill 协作完成任务。

长期上，EA 的核心应与具体入口解耦。命令行、agent skill、桌面界面、网页界面或实验室内部工具都可以成为用户入口；项目设计的稳定部分是工作流、数据模型、证据链和项目记忆。

## 6. 模块化 Skill 架构

EA 长期形态应是父级实验助手 skill 加多个专业子 skill。父级 EA 负责项目上下文、项目记忆、样品/原始数据/报告/图片/provenance 索引和跨 skill 调度。子 skill 负责具体专业能力，例如本地文献库部署、专业绘图、Raman、PL、XRD、IR、SEM/TEM、AFM、电学、电化学和热分析等。

子 skill 应能在两种模式下工作：

- 项目内调用：读取 EA 项目上下文和长期项目记忆，输出可追踪的结果、报告段落、图片和候选记忆。
- 独立调用：完成一次单独任务，但输出格式仍应能被 EA 项目导入。

EA 应包含 `add-skills` 子 skill，负责规范用户加入的新 skill。任何新 skill 要进入 EA 架构，都必须声明输入、输出、review gate、报告格式、图片格式、引用格式、保存位置和 provenance 要求。

详细设计见 `EA_SKILL_MODULE_ARCHITECTURE.md`。

## 7. 核心实体

| 实体 | 作用 |
|---|---|
| Project | 顶层研究空间，包含研究方向、材料体系、项目规则和当前状态 |
| Rule Card | 项目级规则，例如样品编号、质量判断、默认报告语言和数据策略 |
| Experiment | 结构化实验记录，保留原始用户输入和整理结果 |
| Sample | 物理样品或测点集合，连接实验、表征文件和结果 |
| Raw File | 受控保存的原始数据副本及其元数据 |
| Processed Result | 图表、表格、拟合结果、处理后数据和处理元数据 |
| Report | 面向研究者阅读的分析报告或阶段报告 |
| Figure Record | 报告图片、图片 ID、文件路径、原始数据和报告之间的索引 |
| Literature Library | 项目关联文献、PDF/cache、引用条目和检索/下载状态 |
| Skill Module | 可被 EA 调度或单独调用的专业能力模块 |
| Memory Record | 项目记忆中的结构化条目 |
| Review Record | 用户确认、编辑、拒绝或延后的记录 |
| Provenance Record | 输入、输出、脚本、参数、版本和审核关系的记录 |

## 8. 本地项目结构

建议 EA workspace 保持稳定、可读、可迁移：

```text
ea-project/
├── EA_PROJECT.md
├── PROJECT_RULE_CARD.md
├── experiments/
├── samples/
├── raw/
├── processed/
├── figures/
├── reports/
├── literature/
├── skill-registry/
├── reviews/
├── provenance/
├── memory/
├── suggestions/
├── progress/
└── open-items/
```

这些目录本身就是用户和 agent 的共同界面。命名、元数据和引用关系要比“炫技式自动化”更重要。

## 9. 工作流模型

EA 的通用工作流应保持为可组合模块：

1. 创建或打开项目。
2. 接收实验日志、数据文件或研究问题。
3. 识别相关实体和缺失信息。
4. 生成可复核草稿或处理计划。
5. 运行确定性处理或调用专门 skill。
6. 生成图表、表格、元数据和报告草稿。
7. 请求用户对关键解释、决策和记忆写入进行确认。
8. 保存记录、报告、项目记忆和 provenance。
9. 支持后续查询、复盘、比较和版本演化。

不同版本可以选择其中一部分实现，但不应改变这个项目级模型。

## 10. 报告、图片和引用

EA 的科学表达应专业、清晰、可复查：

- 区分观察、计算、处理结果、文献背景、推测、用户确认结论。
- 对不确定性、数据限制和替代解释保持显式说明。
- 报告语言服务于研究者判断，不应为了谨慎而变成言之无物。
- 数据分析应列出被当前数据支持的可能解释，并标明可信度。
- 图表和表格应足够清晰，能让研究者检查数据处理是否合理。
- 图片应有稳定 `figure_id`，能追溯到报告、原始数据、样品和实验条件。
- 报告和对话中引用文献时使用 `[1][2]` 形式，并在文末给出 reference、DOI、本地 PDF 或网页链接。

详细标准见 `EA_REPORT_AND_FIGURE_STANDARD.md`。

## 11. 本地文献库

EA 项目初始化时应检查是否已有本地文献库。若没有，EA 应建议用户创建，并说明文献库会帮助后续分析报告、文献引用、机制解释和项目复盘。

本地文献库应优先复用用户自己的 Zotero 和全文缓存能力。EA 项目保存文献索引、相关性评分、引用条目和缓存链接，而不是把文献管理变成孤立文件夹。若用户没有 Zotero 或不希望启用 Zotero，EA 应提供降级方案，例如只保存 BibTeX、DOI、网页链接和手动导入 PDF 的索引。

详细设计见 `EA_LOCAL_LITERATURE_LIBRARY_SKILL.md`。

## 12. 交互与追问

EA 应及时追问会影响后续项目工作或科学判断的不确定内容。追问应放在回答末尾，便于用户注意和回复。

不影响当前判断的轻微缺失不应打断流程，可以记录为 open item。

## 13. 版本关系

项目设计文件描述 EA 最终要成为怎样的系统。版本计划描述某个阶段先实现什么。

- v0.1 已验证最小纵向闭环和关键边界。
- v0.2 应把 v0.1 从原型内核推进到更稳定、可用、可扩展的研究工作台。
- 后续版本可以继续扩展表征方法、知识库、报告模板、评审机制和用户入口。

版本计划应写在 `EA_V0_2_WORK_PLAN.md` 或后续版本计划文件中，不应覆盖本文件的项目级定位。
