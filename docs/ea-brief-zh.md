## 1. EA 是什么

**实验助手（Experimental Assistant，EA）** 是一个本地优先、人机协作的 AI 研究助手，面向材料科学等实验科学研究者。

一句话概括：*EA 帮助实验科学家组织实验日志、处理研究数据、调用模块化科学技能、生成可追溯的分析报告。*

## 2. EA 不是什么

- 不是完全自主的科学家
- 不仅仅是聊天机器人或脚本集合
- 不是实验室设备控制器（首版不做）
- 不是某个 Claude Code 技能的附属品
- 原始数据绝不能被修改

## 3. 五大核心原则

| # | 原则 | 含义 |
|---|------|------|
| 1 | 人机协作优先 | AI 辅助而非替代研究者，关键步骤需要用户确认 |
| 2 | 以实验为中心 | 基本单位是项目、实验、样品、数据文件，而不仅是聊天消息 |
| 3 | 可追溯性 | 每个重要输出都保留完整的来源链条：文件→技能→参数→结果→决策 |
| 4 | 模块化技能设计 | 科学能力通过可替换的技能模块实现，不绑定特定实现 |
| 5 | 本地优先 | 数据本地存储，原始数据只读，研究者掌握控制权 |

## 4. 领域实体

EA 理解和操作以下核心实体：

| 实体 | 说明 |
|------|------|
| 项目（Project） | 顶层研究项目容器 |
| 实验（Experiment） | 结构化的实验记录卡片 |
| 样品（Sample） | 物理样品，关联材料体系 |
| 材料体系（Material System） | 如 MoS2，用于跨实验分组与分析 |
| 工艺条件（Process Condition） | 温度、时间、气氛等合成/处理参数 |
| 表征文件（Characterization File） | 原始数据文件（CSV/TXT/Excel），只读 |
| 数据处理结果（Data-Processing Result） | 图表、处理后数据、元数据、摘要 |
| 文献记录（Literature Record） | PDF 摘要、文献笔记，可与实验关联 |
| 报告（Report） | Markdown/HTML 报告，含溯源信息 |
| 用户审核决策（User Review Decision） | 在每个审核节点做出的确认/拒绝/编辑 |
| 技能（Skill） | 已注册、可调用的功能单元，有标准化元数据 |

**实体关系：**

```
项目 ──包含──> 实验 ──关联──> 样品 ──属于──> 材料体系
                │
                ├── 工艺条件
                ├── 表征文件 ──产出──> 数据处理结果
                ├── 文献记录
                └── 报告

技能（独立注册） ──被调用后──> 数据处理结果

用户审核决策（贯穿各节点）
  └── 作用于：实验、数据处理结果、报告、文献记录
```

**审核状态流转：** `草稿 → 需审核 → 已确认 / 已拒绝 → 已归档`

## 5. 目标用户与核心痛点

### 目标用户

- **首要用户**：材料科学实验研究者，日常处理实验日志、样品信息、表征数据（拉曼、PL、XRD、SEM/TEM、CVD 生长日志、退火记录等）
- **未来扩展**：化学、物理、环境科学、生物、电化学、器件研究

### 五大痛点

| # | 痛点 | EA 的解决方式 |
|---|------|-------------|
| 1 | 实验记录散落在不同文件夹和格式中，难以检索 | 将非结构化笔记转化为结构化实验记录 |
| 2 | 数据处理重复且缺乏追踪 | 自动化重复步骤，保留处理代码、参数和输出 |
| 3 | AI 输出是黑箱，结论来源不明 | 展示完整工作流溯源 |
| 4 | 文献阅读与实验记录脱节 | 将本地文献笔记与实验数据关联 |
| 5 | 研究记忆脆弱，小观察和失败尝试容易丢失 | 逐步构建项目级研究记忆 |

## 6. 核心工作流（12 步）

| 步骤 | 谁 | 做什么 | 操作渠道 |
|------|-----|--------|---------|
| 1 | 用户 | 创建或打开研究项目 | Web 界面 |
| 2 | 用户 | 输入实验笔记或上传文件 | Web（v0.2+ 支持手机端） |
| 3 | EA | 解析笔记，识别相关实体 | 后端自动 |
| 4 | EA→用户 | 展示提取字段，请求用户确认 | Web（v0.2+ 支持手机推送） |
| 5 | 用户 | 要求 EA 处理关联的数据文件 | Web（v0.2+ 支持手机） |
| 6 | EA | 识别文件类型，建议处理方案 | Web（v0.2+ 支持手机） |
| 7 | 用户 | 确认数据字段和参数 | Web（v0.2+ 支持手机） |
| 8 | EA | 运行相关技能 | 后端自动 |
| 9 | EA | 生成图表、处理后数据和草稿分析 | 后端自动 |
| 10 | EA→用户 | 展示工作流溯源和结果 | Web 完整 / 手机摘要 |
| 11 | 用户 | 确认、编辑或拒绝 AI 解释 | Web（v0.2+ 支持手机） |
| 12 | EA | 将已确认结果写入项目记忆和报告 | 后端自动 |

## 7. 六大强制审核节点

以下节点 EA 必须停下来等待用户确认，不能自动跳过：

| # | 审核节点 | 审核内容 |
|---|---------|---------|
| 1 | 任务计划审核 | 多步工作流执行前展示计划步骤 |
| 2 | 字段审核 | 结构化提取后，展示可编辑字段 |
| 3 | 数据列审核 | 读取文件后，确认 x/y 列含义 |
| 4 | 参数审核 | 确认平滑窗口、基线校正方法、归一化方式等 |
| 5 | 科学解释审核 | 区分：观察结果 / 计算结果 / 文献支持 / 假设 / 已确认结论 |
| 6 | 记忆写入审核 | 保存解释到长期项目记忆前征得用户同意 |

## 8. MVP 范围（v0.1）

### 8.1 MVP 目标

跑通一个完整的闭环：**实验日志输入 → 文件上传 → AI 解析 → 数据处理与绘图 → 草稿分析 → 人工审核 → 报告生成 → 项目记忆更新**。

### 8.2 MVP 四个技能

| 技能 | 输入 | 输出 | 关键行为 |
|------|------|------|---------|
| 实验日志结构化 | 自由文本笔记、可选样品 ID、日期、文件 | 结构化实验卡片（JSON） | 提取实体，等待用户编辑确认 |
| 通用数据读取与绘图 | CSV、TXT、Excel | 图表 PNG、处理后 CSV、元数据 JSON、Markdown 摘要 | 检测列，用户确认 x/y 后绘图 |
| 拉曼光谱分析（演示） | 拉曼光谱文件（CSV/TXT） | 光谱图、峰列表、审慎分析摘要 | 基线校正、检峰、提供多种可能解释 |
| 报告生成 | 已确认的实验日志 + 处理后数据 + 已审核解释 | Markdown 或 HTML 报告（含 9 个部分） | 包含溯源日志 |

### 8.3 v0.1 成功标准（10 条）

1. 能创建本地项目
2. 能接收实验笔记
3. 能提取结构化实验字段
4. 能让用户审核和编辑这些字段
5. 能上传和预览 CSV/Excel/TXT 数据文件
6. 能从用户确认的列生成图表
7. 能运行至少一个材料表征演示技能（优选拉曼）
8. 能生成简短报告
9. 能展示工作流溯源
10. 保存输出但不修改原始数据

### 8.4 v0.1 不做

- 实验室设备控制
- 自动实验决策
- 多用户权限系统
- 云部署
- 大规模 RAG
- 自动在线论文搜索
- 一次做太多表征模块
- 过度复杂的数据库设计
- 工作流跑通前追求 UI 完美
- **手机端接入（微信/飞书）→ 延后到 v0.2+**

## 10. 技术架构

### 10.1 技术选型

| 层级 | 技术 | 选择原因 |
|------|------|---------|
| 前端 | React（Vite） | 桌面应用级体验，组件生态丰富，适合面向多用户的产品 |
| 后端 | FastAPI（Python） | 与 Python 科学生态（numpy, scipy, pandas, matplotlib）原生集成 |
| 存储 | 文件系统 + SQLite | 本地优先，简单，便携 |
| LLM | OpenAI API（GPT 模型，具体版本待项目启动时确定） | 通过适配层可切换；备选 DeepSeek API |
| 技能 | Python 脚本，YAML 元数据注册 | 可扩展，可替换 |

> **注意：** ChatGPT Plus/Pro 订阅不包含 API 额度。API 需要单独注册并预充值，按 token 计费。DeepSeek API 注册即用，价格约 ¥1/百万 token，是低成本备选方案。

### 10.2 架构分层

## 12. 数据与存储原则

1. 原始文件只读
2. 处理后文件写入独立文件夹
3. 每个处理输出必须附带元数据文件
4. 每个 AI 生成的解释必须有审核状态
5. 假设不能被存储为已确认事实（除非用户确认）
6. 报告必须包含溯源信息

**审核状态枚举：** `草稿 | 需审核 | 已确认 | 已拒绝 | 已归档`

## 13. 技能规范

每个技能必须携带 `skill.yaml` 元数据文件：

```yaml
name: raman_analysis
display_name: 拉曼光谱分析
category: data_processing
description: 加载拉曼光谱文件，绘图，检峰，生成审慎分析摘要
input_types: [csv, txt, xlsx]
output_types: [png, csv, markdown]
requires_review: true
risk_level: medium
entrypoint: python skills/raman_analysis/run.py
dependencies: [pandas, numpy, scipy, matplotlib]
```

每个技能必须返回标准结构：

```json
{
  "status": "success",
  "outputs": {
    "figure": "outputs/raman_plot.png",
    "processed_data": "outputs/raman_processed.csv",
    "summary": "outputs/raman_summary.md"
  },
  "parameters": {
    "baseline_correction": "none",
    "normalization": "max_intensity"
  },
  "warnings": ["峰位解释需要用户审核。"],
  "requires_user_review": true
}
```

## 14. 科学安全与可靠性规则

### 行为规则

1. 始终区分：观察 / 计算 / 解释 / 假设 / 结论
2. 不要根据单一数据类型过度断言
3. 证据不强时不能声称因果关系
4. 不能默默混入来自不同材料体系的文献结论
5. 始终保留原始数据
6. 解释模糊时展示不确定性
7. 写入长期项目记忆前必须获得用户确认

### 语言规范

**推荐使用：**
- "可能表明"
- "与……一致"
- "可能与……相关"
- "需要进一步确认"

**禁止使用：**
- "这证明了 X"
- "这确认了 X"
- "这个机制是确定的"

## 15. AI 开发协作原则

1. 始终保留上述产品哲学
2. 仅在必要时提出澄清性问题
3. 偏好小而可测试的实现步骤
4. 保持原始数据只读
5. 每生成一个文件就要添加元数据和溯源
6. 避免早期过度架构设计
7. 维护可读的文档
8. 创建示例数据和示例工作流
9. 在科学解释选择上保持用户参与
10. 将 EA 视为研究产品，不仅是编程练习

## 16. 关键决策记录

- **手机端接入**（微信/飞书）延后至 v0.2+
- **MVP 跑通闭环**，但架构从第一天起要支持无限技能扩展（XRD、AFM、PL、SEM 等）
- **最终产品面向真实用户**，不是个人工具——选 React + FastAPI
- **LLM 供应商**：首版用 OpenAI API（GPT 模型，具体版本待项目启动时确定），备选 DeepSeek API；通过适配层实现可切换
